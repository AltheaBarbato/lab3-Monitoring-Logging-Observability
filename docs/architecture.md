# Monitoring Architecture
**Name:** Althea Barbato

## What's deployed

Four tools running as Docker containers on `webserver01` (163.192.117.50, Oracle Cloud Always Free):

| Tool | Port | What it does |
|---|---|---|
| Prometheus | 9090 | Scrapes metrics from all exporters every 15s, evaluates alert rules |
| Grafana | 3000 | Dashboards and alerting UI on top of Prometheus |
| Uptime Kuma | 3001 | Service availability monitoring with visual status page |
| nginx_exporter | 9113 | Translates nginx's stub_status page into Prometheus metrics |

Plus these already running from earlier labs:
- **node_exporter** (9100) — system metrics (CPU, RAM, disk, network, systemd units)
- **nginx** (80) — the actual web server being monitored
- **fail2ban** — blocks IPs with repeated SSH failures, active bans show up in auth.log

## Data flow

```
[nginx]  -->  [nginx stub_status :8080]  -->  [nginx_exporter :9113]  -->┐
[system] -->  [node_exporter :9100] ---------------------------------->---┤
              [node_exporter textfile_collector] <-- [cron: count auth.log] ┘
                                                                          │
                                                     [Prometheus :9090] <─┘
                                                            │
                                                     evaluates alert rules
                                                            │
                                                     [Grafana :3000] <── reads metrics + alert state
                                                     [Uptime Kuma :3001] <── polls HTTP endpoints
```

Prometheus pulls (scrapes) from all exporters. Nothing pushes to it. Alert rules live in `/etc/prometheus/alert.rules.yml` — Prometheus evaluates them on every scrape cycle and sets alert state (inactive / pending / firing). Grafana reads from Prometheus as its data source and mirrors the alert state in its own alerting UI.

The failed login metric is generated by a cron job that runs every minute, counts "Failed password" lines in `/var/log/auth.log`, and writes a `.prom` file to `/var/lib/node_exporter/textfile_collector/`. node_exporter reads that directory automatically and exposes it as `node_failed_ssh_logins_total`.

## Metrics collected

**System (via node_exporter):**
- CPU utilization — `node_cpu_seconds_total`
- Memory — `node_memory_MemAvailable_bytes`, `node_memory_MemTotal_bytes`
- Disk — `node_filesystem_avail_bytes`, `node_filesystem_size_bytes`
- Network — `node_network_receive_bytes_total`, `node_network_transmit_bytes_total`
- Systemd unit state — `node_systemd_unit_state` (includes fail2ban, docker, nginx)

**NGINX (via nginx_exporter):**
- `nginx_http_requests_total`
- `nginx_connections_active`
- `nginx_connections_waiting`, `nginx_connections_reading`, `nginx_connections_writing`

**Security (via textfile collector):**
- `node_failed_ssh_logins_total` — cumulative count of failed SSH login attempts from auth.log

**Availability (Prometheus self-scrape):**
- `up` — 1 if target was reachable on last scrape, 0 if not
- `scrape_duration_seconds` — how long each scrape took

## Alert rules

| Alert | Condition | For | Severity |
|---|---|---|---|
| InstanceDown | `up == 0` | 30s | critical |
| HighCPUUsage | CPU > 80% | 1m | warning |
| LowDiskSpace | root < 15% free | 2m | warning |
| HighMemoryUsage | memory > 85% | 2m | warning |
| FailedSSHLogins | > 20 new failures in 1h | immediate | warning |

The `for:` field prevents one-off blips from firing alerts — the condition has to hold continuously for the specified duration before the alert moves from Pending to Firing. InstanceDown is 30s because that's the most critical; disk and memory get 2 minutes to filter out spikes.
